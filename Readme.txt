1. Текст задачи

Написать программу, которая приводит уравнение к каноническому виду: приводит подобные слагаемые и приравнивает сумму к нулю.
 
Уравнение может быть любого порядка, может содержать любое количество переменных, может быть записано со скобками (в этом случае приложение должно раскрыть скобки).
 
Уравнение на вход будет передано в виде строки в следующем формате:
P1 + P2 + ... = ... + PN
где P1..PN - слагаемые, представленные в виде:
ax^k
где a - число с плавающей точкой;
k - целое число;
x - переменная (переменных у одного слагаемого может быть несколько).
 
Например, может быть дано уравнение следующего вида:
x^2 + 3.5xy + y = y^2 - xy + y
Оно должно быть приведено к виду:
x^2 - y^2 + 4.5xy = 0
 
Программа должна поддерживать два режима работы – интерактивный и файловый.
В интерактивном режиме программа предлагает ввести выражение и выводит результат по окончании ввода. После этого приглашение ввести выражение повторяется. Завершить работу можно с помощью сочетания Ctrl-C.
В файловом режиме программа принимает в качестве параметра имя файла содержащего строчки с выражениями и записывает результат работы в новый файл с добавлением расширения .out.
 
Будет плюсом, если вы напишете юнит-тесты.

Задание рассчитано на 4-5 часов.


2. Комментарии по получившейся реализации

- Качественно выполнить задание за 4-5 часов имхо малореально. Под качественно я имею в виду: хорошее покрытие юнит-тестами, вылавливание всех хитрых краевых случаев при парсинге, хорошо струтурированный и качественный код, комментарии во всех нужных местах.

- Я потратил на задачу больше 7 часов и все равно не все успел. Поэтому в моем решении нет ни консольного ни файлового режима - только парсинг и бизнес-логика связанная с каноническим видом уравнений и многочленов. В Program.cs просто пример использования классов приведения уравнения.

- Однако в корректности работы кода легко убедится по большому количеству юнит-тестов ко всем важным классам проекта. При написании я использовал TDD.

- Опять-таки в силу нехватки времени, я уверен, что есть баги в коде парсеров для хитрых краевых случаев. Также я не стал заморачиватся с излишним в этой задаче ООП и добавлять интерфейсы для сигнатур классов-парсеров.

- В задаче нет разъяснений о трактовке записи слагаемоего  вида "2xyz^3" - я трактую это как 2*(x*y*z)^3 а не как "2*x*y*(z^3)".

- Прорамма не раскрывает скобки во входных данных, а просто вырезает их. Операция сложения коммутативна и ассоциативна, поэтому для задачи скобки не играют никакой роли.
